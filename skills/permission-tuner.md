# Skill: permission-tuner

> Analyze permission denial patterns and generate optimal allow/deny rules.
> Inspired by: pro-workflow permission-tuner

## When to Trigger

- User says "tune permissions", "why was that denied", "permission analysis"
- After repeated permission denials slow down a workflow
- When setting up a new project and configuring agent permissions
- When a user reports that the agent is too restricted or too permissive

## Workflow

### Step 1 — Collect Denial History

Scan `.memory/failures.md` and session context for permission-related events:

- Tool calls that were blocked or denied
- Operations that required confirmation and were approved
- Operations that required confirmation and were rejected
- Sandboxed commands that failed due to restrictions

Build a denial log:

| # | Timestamp | Operation | Reason | User Action |
|---|-----------|-----------|--------|-------------|
| 1 | 10:23 | `rm -rf dist/` | destructive command | approved |
| 2 | 10:25 | `npm publish` | network access | denied |
| 3 | 10:30 | `git push --force` | destructive git op | denied |
| 4 | 10:45 | `rm -rf dist/` | destructive command | approved |

### Step 2 — Analyze Patterns

Classify denials into categories:

- **Appropriate blocks** — genuinely dangerous operations correctly caught
- **Overly restrictive** — safe operations blocked by broad rules (e.g., all file deletes blocked when only `dist/` cleanup is routine)
- **Repetitive approvals** — same operation approved every time, should be auto-allowed
- **Missing rules** — dangerous operations that SHOULD be blocked but aren't

For each pattern, calculate:
- Frequency (how often it occurs per session)
- User override rate (how often the user approves after a block)
- Risk level (what's the worst case if this operation goes wrong)

### Step 3 — Generate Rules

Produce a set of suggested permission rules:

```yaml
# Suggested permission rules
# Generated by permission-tuner — {date}

allow:
  - pattern: "rm -rf dist/"
    reason: "Routine build cleanup — approved 12/12 times"
    risk: LOW

  - pattern: "npm run *"
    reason: "Package scripts — never denied"
    risk: LOW

deny:
  - pattern: "git push --force *"
    reason: "Force push blocked — denied 5/5 times"
    risk: HIGH

  - pattern: "npm publish"
    reason: "Publish blocked — requires explicit user intent"
    risk: HIGH

prompt:
  - pattern: "rm -rf {path}"
    condition: "path is not dist/ or build/ or node_modules/"
    reason: "Non-standard directory deletion — confirm each time"
    risk: MEDIUM
```

### Step 4 — Present Recommendations

Show the user a clear summary:

- Rules that would REDUCE friction (auto-allow safe repetitive operations)
- Rules that would INCREASE safety (auto-deny dangerous patterns)
- Rules that should stay as PROMPT (case-by-case decisions)

### Step 5 — Apply Rules

If the user approves, write rules to the appropriate config:
- `.claude/settings.json` for Claude Code
- `.codex/config.yaml` for Codex CLI
- Copilot CLI permission settings
- Or a universal `.permissions.yml` that skills can reference

## Output Format

```markdown
## Permission Analysis

**Period:** {date range or session count}
**Total events:** {N} denials, {N} approvals, {N} auto-allowed

### Friction Points (should auto-allow)
| Operation | Frequency | Override Rate | Suggested Rule |
|-----------|-----------|---------------|----------------|
| `rm -rf dist/` | 12/session | 100% approved | ALLOW |
| `npm run build` | 8/session | 100% approved | ALLOW |

### Safety Gaps (should auto-deny)
| Operation | Risk | Current | Suggested Rule |
|-----------|------|---------|----------------|
| `git push --force` | HIGH | prompted | DENY |

### Suggested Rules File
{yaml output as above}

### Impact Estimate
- Prompts eliminated per session: ~{N}
- New safety blocks added: {N}
- Net friction reduction: {percentage}
```

## Rules

- NEVER auto-allow destructive operations without explicit user approval of the rule
- Permission analysis is READ-ONLY by default. Only write rules if user says "apply"
- High-risk operations (publish, deploy, force-push, delete) default to DENY or PROMPT, never ALLOW
- Track both denials AND approvals — a 100% approval rate on a prompted operation means it should be auto-allowed
- Rules should be as specific as possible. "allow rm" is too broad; "allow rm -rf dist/" is correct
- Respect the principle of least privilege — when in doubt, keep the prompt
- Generate rules in the format appropriate for the active CLI, not a generic format

## Not Responsible For

- Enforcing permissions at runtime (that's the CLI's job)
- System-level file permissions (chmod, ACLs)
- Git branch protection rules (those are server-side)
- Authentication or authorization for external services
