# Skill: wiki

> Auto-maintained project knowledge base. Reduces token usage by caching codebase knowledge as readable markdown.
> Inspired by: oh-my-claudecode (omx wiki), oh-my-codex (.omx/wiki/)

## When to Trigger

- User says "update wiki", "generate docs", "what does this project do"
- At session start: load wiki overview for quick context (if `.wiki/` exists)
- After significant codebase changes: refresh affected wiki pages
- User says "wiki list", "wiki query {term}", "wiki refresh"

## Workflow

### Step 1 — Check Existing Wiki

Check if `.wiki/` exists at the project root.
- If yes, read `.wiki/index.md` for current state
- If no, create `.wiki/` and generate initial pages from codebase analysis

### Step 2 — Scan Codebase

Analyze the codebase to gather content for wiki pages:
- **Architecture:** Read entry points, config files, directory structure. Identify layers (API, services, data, UI).
- **Module index:** Glob for source directories and key files. Summarize each module's purpose.
- **API reference:** Find route definitions, exported functions, public interfaces.
- **Conventions:** Look for linter configs, existing CONTRIBUTING.md, code patterns that repeat.
- **Glossary:** Extract domain terms from code comments, README, and variable names.

### Step 3 — Generate/Update Wiki Pages

Create or update the following pages in `.wiki/`:

```
.wiki/
  index.md           # Table of contents with links to all pages
  architecture.md    # System architecture: layers, data flow, key components
  modules.md         # Module index: purpose of each directory/package
  api.md             # API surface: routes, functions, interfaces
  conventions.md     # Project conventions: naming, patterns, tooling
  glossary.md        # Domain terms and definitions
  {custom}.md        # User-created pages (preserved across refreshes)
```

Each page follows this format:
```markdown
# {Page Title}

> Last updated: {YYYY-MM-DD}
> Source: auto-generated by wiki skill

{content}
```

### Step 4 — Lint & Validate

After generating pages:
- Check for broken internal links between wiki pages
- Verify referenced files/paths still exist in the codebase
- Flag outdated content (references to deleted files or changed APIs)
- Update `index.md` with any new pages

## Output Format

```markdown
## Wiki: {action performed}

**Pages updated:** {count}
**Pages created:** {list}
**Stale content found:** {count, with details}

### Index
- architecture.md — {1-line summary}
- modules.md — {1-line summary}
- api.md — {1-line summary}
- conventions.md — {1-line summary}
- glossary.md — {1-line summary}
```

## Rules

- Wiki is markdown-first, search-first (grep over files) — no vector DB or fancy indexing
- When exploring unfamiliar code, check wiki FIRST before reading source files
- Never delete user-created pages during refresh — only update auto-generated ones
- Keep pages concise: architecture overview should be 1-2 pages, not a novel
- `.wiki/` should be gitignored by default (agent-local context). If user wants shared docs, move to `docs/wiki/` instead.
- Re-generate stale pages when underlying code changes, not on every session start
- Auto-update integrates with doc-maintainer skill: when doc-maintainer detects code changes, trigger wiki refresh for affected pages

## Not Responsible For

- User-facing documentation (see doc-maintainer skill)
- API documentation generation tools (Swagger, JSDoc — wiki supplements, doesn't replace)
- Code review or analysis (see review skill)
- Memory/context persistence (see remember skill — wiki is project docs, remember is session learnings)
